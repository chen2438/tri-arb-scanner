# Tri-Arb Scanner Agent 协作约定

- `DOCS.md` 是唯一权威的功能与架构文档。任何影响系统行为、接口、配置、验证方式、交易所语义或
  安全边界的变更，都必须在同一次工作中同步更新 `DOCS.md`。`README.md` 只保留项目入口信息，
  不与 `DOCS.md` 重复维护完整规格。
- 当前产品边界是**只读扫描器**。在 `DOCS.md` 明确升级安全边界且用户明确授权前，不得实现或调用
  下单、撤单、划转、提现等私有交易能力，也不得要求交易权限 API Key。
- MEXC 是首个且当前优先支持的交易所。交易所协议、交易规则、费率或限频相关实现必须以官方文档
  和可复现响应样本为依据；MEXC 特有语义应封装在适配器边界内，核心套利计算不得依赖交易所字段名。
- 每个可独立验收的功能使用一个单独的 Git 提交；若用户明确要求本次不提交或不推送，以用户要求为准。
- 每条提交信息必须包含：
  1. 简洁的 Conventional Commit 风格标题；
  2. 一个空行，以及说明“改了什么、为什么改”的有意义 description；
  3. 实现该变更的 Agent 共同作者 trailer。
- Codex Agent 提交必须使用当前实际模型名而非产品名，例如
  `Co-authored-by: GPT-5.6 Sol <noreply@openai.com>`；Claude Code 提交必须使用当前适用的
  Anthropic 共同作者 trailer。
- 完全由用户本人实现且没有 Agent 参与的提交可以改以 `Human-authored: true` 结尾；Agent 禁止
  使用或建议冒用该标记来绕过共同作者要求。
- 本地仓库必须使用 `git config core.hooksPath .githooks` 启用版本化 `commit-msg` hook。提交后、
  推送前必须执行 `python3 scripts/check_commit_messages.py --commit HEAD` 再次验证 Git 实际解析的
  message。不得用字面量 `\\n` 拼接提交正文或 trailer。
- 禁止只有标题、没有正文的提交。当交易所兼容性、数值精度、安全行为或验证结果对后续维护有帮助时，
  必须在 description 中记录。
- 修改 `.env.example` 时，必须在同一次工作中把新增变量同步补入本地 `.env`，但不得覆盖 `.env`
  中任何已有值；不得读取、输出、暂存或提交 `.env` 及其中的凭据。
- 金额、价格、数量、费率和收益率计算必须使用十进制定点语义，禁止用二进制浮点数承担业务计算。
  路径收益必须计入交易方向、买一/卖一价、手续费、交易规则、数量取整、最小下单约束和可用深度；
  不能把仅基于最新成交价的理论价差标为“可执行机会”。
- WebSocket/REST 数据必须带来源时间与本地接收时间；过期、乱序、缺腿或重连恢复期间的数据不得产生
  可执行机会。对应边界必须有确定性测试。
- 提交前执行 `DOCS.md` 中与改动相关的检查。仓库级 Python 命令从仓库根目录运行；若以后增加
  `frontend/`，只有前端包管理命令进入该目录，前端变更还必须完成浏览器验证。
- 用户授权提交和推送时，每次提交通过校验后立即推送当前分支到其远端上游；若推送失败，必须明确
  告知失败原因，不得把仅存在于本地的提交描述为已交付。
